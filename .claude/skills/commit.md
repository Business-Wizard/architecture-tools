Inspect uncommitted changes, determine the appropriate conventional commit message, stage files, and commit.

## Workflow

1. **Inspect changes:**
   - Run `git status` to see all changes (staged and unstaged)
   - Run `git diff --cached` for staged changes
   - Run `git diff` for unstaged changes

2. **Analyze and group:**
   - Identify the primary change type (feat, fix, refactor, docs, etc.)
   - Determine the scope (which area/module)
   - Group related changes into a single logical commit

3. **Generate message:**
   - Write a concise subject line (max 50 chars)
   - Add body if the diff warrants explanation (separated by blank line)
   - Follow Conventional Commits format strictly

4. **Stage and commit:**
   - Stage all relevant files: `git add <files>`
   - Commit: `git commit -m "<message>"`
   - Show result: `git log --oneline -1`

## Conventional Commits Rules

**Subject line:**
- Format: `<type>(<scope>): <description>`
- Types: `feat`, `fix`, `refactor`, `perf`, `test`, `docs`, `build`, `ci`, `chore`
- Imperative mood: "add" not "added" or "adds"
- No trailing period
- Max 50 characters (GitHub constraint)
- Hard limit: 72 characters
- Lowercase after colon

**Body (only if needed):**
- Blank line separates subject from body
- Explain *what* and *why*, not *how*
- Wrap at 72 characters
- Name specific files, functions, or systems

**Prohibited:**
- NO attribution (Co-Authored-By, Generated by, AI, Claude, GPT)
- NO filler phrases ("this commit", "update code", "make changes")
- NO vague subjects ("fix bug", "refactor stuff")

## Example Outputs

**Simple feature (no body needed):**
```
feat(report): apply UTF8_FULL table preset for formatting
```

**More complex change (needs body):**
```
feat(cli): add verifiers field to config

Allow users to specify which verifiers (ruff, basedpyright, pytest)
to enable in awt.toml. Baseline checks and mutation runs now only
execute enabled verifiers, skipping tools with pre-existing errors.
```

## Important

- Only stage and commit changes relevant to a single logical unit
- If multiple unrelated changes exist, ask the user which to commit
- Do NOT push to remote
- Do NOT amend existing commits
